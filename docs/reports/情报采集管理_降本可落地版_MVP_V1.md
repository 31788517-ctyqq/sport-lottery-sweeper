# 情报采集管理（降本可落地版 MVP）
更新时间：2026-02-19

## 1. 目标
用最小成本在 4-6 周内打通“采集 -> 结构化 -> 评分 -> 推送（钉钉）”闭环，先可用、可观测、可迭代。

## 2. MVP 范围（强约束）
1. 数据源仅 3 个：`500W`、`天天盈球`、`1个新闻源`（如腾讯体育）。
2. 情报类型仅 5 类：伤病、天气、战意、胜平负、让球胜平负。
3. 采集频率只做固定时点：`24h/12h/6h/1h`。
4. 推送仅文本卡片（钉钉），PDF 放到二期。
5. 用户配置仅 4 项：联赛、球队、风险偏好、推送频率。

## 3. 非目标（MVP 不做）
1. 全网 10+ 数据源全接入。
2. 比分/半全场/总进球完整预测链路。
3. 变化触发推送自动上线（先观测后启用）。
4. 可视化复杂报表与重型 AI 推理。

## 4. 方案设计
## 4.1 架构
1. 一个微服务代码仓：`intelligence-collection-service`。
2. 三类进程：`api`、`collector-worker`、`scheduler`。
3. Redis 队列 + MySQL 持久化。
4. 单镜像多进程可部署（先降运维成本），保留后续拆分能力。

## 4.2 评分与降噪（规则版）
1. `confidence = 来源权重 + 一致性 + 时效性`（可配置）。
2. 默认推送门槛：`confidence >= 0.72` 且有效来源 >= 2。
3. 同场比赛每天最多 1 条主推。
4. 不达标状态统一为“观察中”。

## 5. 交付内容
1. 采集任务管理页（创建、重试、取消、日志）。
2. 比赛情报结果页（来源观点、时间、链接、置信度）。
3. 用户订阅轻配置页。
4. 钉钉绑定与测试发送。
5. 运维看板指标：任务成功率、解析失败率、推送送达率、队列积压。

## 6. 成本评估（相对）
1. 开发成本：中（后端为主，前端中等）。
2. 运维成本：低-中（单服务、少容器）。
3. 维护成本：中（3个源站解析维护）。

## 7. 卡点与应对
1. 反爬/页面改版：为每个源站做解析器版本号 + 快速禁用开关。
2. 任务重跑重复数据：幂等键 `match_id + source + type + planned_at`。
3. 推送噪声：默认节流 + “观察中”兜底。
4. 钉钉限流：指数退避重试 + 死信队列。

## 8. 分期计划
### 里程碑 M1（第 1-2 周）
1. 表结构、任务系统、基础采集器。
2. 固定时点调度与日志链路。

### 里程碑 M2（第 3-4 周）
1. 评分规则与推送门槛。
2. 钉钉发送与订阅配置。
3. 预发布联调与回归。

### 里程碑 M3（第 5-6 周，可选）
1. 增加第 4/5 个情报类型。
2. 增加来源质量统计与运营报表。

## 9. 上线验收
1. 任务成功率 >= 90%。
2. 推送送达率 >= 98%。
3. 同场重复推送率 < 2%。
4. 人工干预率 < 20%。

## 10. 未来升级建议（从 MVP 过渡）
1. 增加 `scoring-worker` 独立进程，支持动态权重。
2. 上线“变化触发推送”但先灰度 10% 用户。
3. 引入 PDF Worker，先支持标准模板。
4. 源站接入改为插件机制（统一接口 + 每源站独立 parser 包）。

# 情报采集管理（全功能版 目标态）
更新时间：2026-02-19

## 1. 目标
建设可持续演进的情报中台：多源采集、多时点追踪、多策略评分、个性化推送、钉钉与 PDF 交付、赛后复盘闭环。

## 2. 全功能范围
1. 数据源：10+（体育站点 + 社媒 + 内容平台），插件化接入。
2. 情报类型：场外信息 9 类 + 预测信息 5 类全覆盖。
3. 采集模式：固定时点 + 变化触发 + 人工补采。
4. 推送渠道：钉钉（个人/群）+ 站内消息（预留）。
5. 交付形态：文本卡片 + 模板化 PDF（简版/标准版/复盘版）。

## 3. 核心能力设计
## 3.1 多层评分
1. 置信度 `confidence`：来源权重、一致性、时效性、历史命中。
2. 稳定性 `stability_score`：多时点趋势与反转检测。
3. 风险等级 `risk_level`：冲突程度、信息缺失、样本稀疏度。
4. 可解释证据：最多 3 条关键证据 + 完整证据树。

## 3.2 个性化引擎
1. 用户订阅画像：联赛/球队/玩法/风险偏好/信息密度。
2. 推送节流：单场上限、日上限、冷却窗口。
3. 反馈学习：有用/无用/过载信号回流策略。

## 3.3 复盘闭环
1. 赛后对账：结论 vs 实际结果。
2. 来源质量更新：命中率、可用率、时效分。
3. 阈值自适应：按联赛、玩法、用户群分层校准。

## 4. 微服务与容器化（目标态）
1. `intelligence-api`：外部接口与权限。
2. `intelligence-collector`：源站采集与解析。
3. `intelligence-scoring`：评分与阈值决策。
4. `intelligence-push`：消息编排、节流、送达回执。
5. `intelligence-pdf`：报告渲染与存储。
6. `intelligence-scheduler`：时间任务与补偿任务。
7. `intelligence-ops`：健康检查、监控、审计与告警。

容器化原则：
1. 每服务独立镜像、独立扩缩容。
2. K8s 部署，按队列与 CPU 自动扩容。
3. 配置中心 + Secret 管理敏感信息。
4. 全链路追踪（TraceID）与统一日志。

## 5. 全量接口域（建议）
1. 采集域：任务、计划、重试、日志、源站健康。
2. 结果域：比赛情报、时点轨迹、证据树、冲突详情。
3. 评分域：评分结果、阈值配置、来源权重。
4. 订阅域：用户模板、偏好、频率、分组。
5. 渠道域：钉钉绑定、测试、发送、回执。
6. 报告域：PDF 渲染、下载、归档、二次发送。
7. 复盘域：赛后分析、策略效果、来源榜单。

## 6. 数据模型（目标态新增）
1. `intel_signal_snapshot`：每时点快照（趋势必需）。
2. `intel_source_quality`：来源质量评分与权重。
3. `intel_user_subscription`：用户订阅偏好。
4. `intel_channel_binding`：钉钉渠道绑定。
5. `intel_push_delivery_log`：送达链路日志。
6. `intel_pdf_report`：报告渲染与文件元数据。
7. `intel_feedback_event`：用户反馈事件。
8. `intel_postmatch_review`：赛后复盘结果。

## 7. 成本与收益评估
1. 开发成本：高（多服务 + 算法 + 前后端联动）。
2. 运维成本：中高（K8s + 监控 + 队列治理）。
3. 收益：高（准确率提升、留存提升、可规模化运营）。

## 8. 主要卡点与治理建议
1. 源站稳定性：解析器插件化 + 合约测试 + 自动回归抓取。
2. 算法可信度：先规则后学习，强制可解释输出。
3. 推送打扰：严格节流 + 反馈闭环 + 用户可控开关。
4. 渠道稳定性：幂等投递、死信队列、补偿重试。
5. PDF 性能：异步渲染池 + 模板版本控制 + 缓存。
6. 数据治理：质量分层、异常探测、手工兜底入口。

## 9. 路线图（建议）
### Phase 1（0-2 个月）
1. MVP 跑通（采集、评分、钉钉推送、基础订阅）。
2. 关键指标上线：成功率、送达率、噪声率。

### Phase 2（2-4 个月）
1. 接入更多来源、上线 PDF 标准版。
2. 上线变化触发推送（灰度）。
3. 上线赛后复盘仪表盘。

### Phase 3（4-8 个月）
1. 分联赛阈值策略与动态权重。
2. 更细粒度个性化推荐。
3. 多渠道分发与运营自动化。

## 10. 管理与实施建议
1. 建立“产品-算法-爬虫-运维”联合评审机制，每周复盘。
2. 所有策略参数配置化（不写死代码）。
3. 建立“灰度-监控-回滚”三件套作为上线门禁。
4. 先打通业务闭环，再追求复杂模型，避免过度设计。

# 日志系统改造开发计划

## 文档信息
- **版本**: 1.0
- **创建日期**: 2026-02-10
- **作者**: AI开发团队 (coder1)
- **状态**: 草案

## 1. 项目背景

当前竞彩足球扫盘系统的日志模块存在多个功能性问题和架构缺陷，影响了系统的可维护性和用户体验。通过对现有日志系统的全面评估，发现以下主要问题：

### 1.1 当前问题概述
- **前端数据处理逻辑错误**: 多个日志页面存在 `response.data` vs `response` 的混淆问题
- **变量定义不一致**: 部分页面缺少必要的响应式变量定义
- **功能缺失**: 缺少日志导出、实时推送、高级筛选等关键功能
- **性能瓶颈**: 大数据量场景下缺乏优化措施
- **用户体验不佳**: 页面重复、操作反馈不足、移动端适配缺失

### 1.2 业务影响
- 日志无法正常显示，影响运维监控
- 缺少数据分析能力，无法满足审计需求
- 系统稳定性受影响，存在运行时错误风险

## 2. 改造目标

### 2.1 核心目标
- **修复现有bug**: 确保所有日志页面能正常加载和显示数据
- **统一架构**: 建立标准化的日志处理流程和组件结构
- **增强功能**: 添加导出、实时监控、高级筛选等企业级功能
- **提升性能**: 优化大数据量处理和查询性能
- **改善体验**: 提供一致的用户界面和操作体验

### 2.2 成功指标
- 所有日志页面100%正常显示数据
- 日志查询响应时间 < 2秒（10万条数据内）
- 支持CSV/Excel格式导出
- 实现实时日志推送功能
- 移动端适配率达到100%

## 3. 系统架构设计

### 3.1 整体架构
```
┌─────────────────┐    ┌─────────────────┐    ┌─────────────────┐
│   日志生产者     │───▶│   日志存储层     │───▶│   日志消费层     │
│ (应用/服务/中间件) │    │ (LogEntry模型)   │    │ (前端/分析/告警)  │
└─────────────────┘    └─────────────────┘    └─────────────────┘
```

### 3.2 数据模型
**核心表**: `log_entries`
- **字段设计**: 
  - `id`: 主键
  - `timestamp`: 时间戳 (索引)
  - `level`: 日志级别 (DEBUG/INFO/WARN/ERROR/CRITICAL)
  - `log_type`: 日志类型 (system/user/security/api)
  - `module`: 模块名称
  - `message`: 日志消息
  - `user_id`: 用户ID (可为空)
  - `ip_address`: IP地址
  - `request_path`: 请求路径
  - `response_status`: 响应状态码
  - `duration_ms`: 请求耗时
  - `extra_data`: 额外JSON数据
  - `created_at`: 创建时间

**索引策略**:
- `idx_timestamp_level`: 时间戳 + 级别
- `idx_user_timestamp`: 用户ID + 时间戳  
- `idx_module_level`: 模块 + 级别
- `idx_log_type`: 日志类型

### 3.3 API接口设计
**基础路由**: `/api/v1/admin/system/logs/db/{log_type}`

| 端点 | 方法 | 描述 |
|------|------|------|
| `/logs/db/{type}` | GET | 获取指定类型日志列表 |
| `/logs/db/{type}/count` | GET | 获取指定类型日志总数 |
| `/logs/db/statistics` | GET | 获取日志统计信息 |
| `/logs/db/export` | GET | 导出日志数据 |
| `/logs/db/search` | GET | 高级搜索日志 |

### 3.4 前端组件架构
```
LogManagement.vue (主入口)
├── SystemLogs.vue (系统日志)
├── UserLogs.vue (用户日志)  
├── SecurityLogs.vue (安全日志)
├── APILogs.vue (API日志)
└── LogUtils.js (工具函数)
```

## 5. 技术栈与工具

### 5.1 前端技术栈
- **框架**: Vue 3 + Composition API
- **UI库**: Element Plus
- **状态管理**: Pinia
- **图表库**: ECharts
- **HTTP客户端**: Axios + 自定义拦截器
- **构建工具**: Vite

### 5.2 后端技术栈
- **框架**: FastAPI
- **数据库**: PostgreSQL/SQLite (SQLAlchemy ORM)
- **缓存**: Redis
- **消息队列**: Celery (可选)
- **WebSocket**: Starlette WebSocket

### 5.3 开发工具
- **代码格式化**: Prettier + ESLint
- **测试框架**: Vitest (前端), Pytest (后端)
- **CI/CD**: GitHub Actions
- **文档**: Markdown + Mermaid

## 7. 风险评估与应对

### 7.1 技术风险
| 风险 | 影响 | 应对措施 |
|------|------|----------|
| WebSocket兼容性问题 | 实时推送功能受限 | 提供轮询降级方案 |
| 大数据量性能瓶颈 | 查询响应慢 | 分页+虚拟滚动+缓存 |
| 敏感信息过滤遗漏 | 安全漏洞 | 多重过滤+人工审核 |

### 7.2 项目风险
| 风险 | 影响 | 应对措施 |
|------|------|----------|
| 需求变更 | 开发延期 | 采用敏捷开发，分阶段交付 |
| 资源不足 | 进度延迟 | 优先实现核心功能 |
| 兼容性问题 | 功能异常 | 充分测试+回滚机制 |

# 平局预测管理（形态B）代码结构对齐实施方案（CRAFT）

## 1. 目标与范围

### 1.1 本期目标
- 实现**形态 B：下注建议 + Kill-Switch**（不自动下单）。
- 建议输出包含：`decision(BET/SKIP)`、`stake_pct(0~1)`、建议时机窗口、原因标签。
- Kill-Switch 状态：`RUN/WARN/STOP`，当 `STOP` 时强制不输出建议（返回 `SKIP`）。
- **核心要求：本期实施不属于修补，而是对平局预测核心模块进行“代码重写级”的架构升级，确保逻辑清晰、前后端分离彻底。**

### 1.2 已冻结业务口径
- `fee_rate`、`rebate_rate`：可配置，默认 `0.08`、`0.75`。
- 返还规则：**仅命中返**，且按 `stake` 返（`on_win_stake`）。
- 建议规则：`R1`（`edge = base_prob - implied_prob` + 完整度过滤 + 时机窗口）。
- `implied_prob = 1 / odds_draw`（方案 A，不做返还率校正）。
- `edge_min = 0.03`，`edge_full = 0.10`。
- `stake_pct = clamp((edge-edge_min)/(edge_full-edge_min), 0, 1)`。
- 窗口默认：`global = 2~6h`（需严格约束取值时间）。
- Kill-Switch：采用行业默认并保留 `WARN` 档。

### 1.3 v1 硬门槛（缺任一项即 `SKIP`）
1. `base_prob`
2. `odds_draw_place`
3. `kickoff_time`
4. `league_key`
5. `match_id`

## 2. 与当前代码结构对齐原则

### 2.1 后端路由不拆域，沿用现有入口
- 统一入口：`backend/api/v1/draw_prediction.py`（`prefix=/draw-prediction`）。
- 在该文件下扩展 `draw-suggestion/*` 子路径，避免新增平级路由。

### 2.2 异步任务机制升级
- 当前有内存任务：`_FETCH_TASKS + /tasks/{task_id}`。
- v1 升级为**轻量持久化**（`async_tasks` 表），避免重启丢任务。

### 2.3 配置系统复用 `SystemConfig`
- 复用现有 `SystemConfig`。
- Kill-Switch 阈值支持**热加载**，无需重启。

### 2.4 前端 API 保持统一
- 继续在 `frontend/src/api/drawPrediction.js` 追加接口，减少分散。

### 2.5 重构与代码质量原则
1. **Domain Logic Isolation**
   - Controller 禁止写 SQL/业务计算。
   - Service 拆分：`Calculator`（纯计算）+ `Service`（I/O）。
2. **统一状态管理**
   - Vue 页面禁止堆业务逻辑。
   - API/轮询/错误态必须封装在 `composables/*`。
3. **Anti-Corruption Layer**
   - 外部数据必须先经 `Adapter` 清洗为标准对象后再计算。

## 3. 当前代码现状与关键差距

### 3.1 已有能力
1. 平局预测主路由、AI 扫盘、Poisson 扫盘、训练/监控页面已存在。
2. `SystemConfig` 可用，支持 JSON 配置。

### 3.2 关键差距（必须补）
1. 赛果回写仍是随机模拟。
2. 缺少 500 平局赔率快照表与采集逻辑。
3. 缺少建议表、paper bet 表、Kill-Switch 状态表。
4. 缺少外部 ID 映射机制（`match_id <-> 500 fixture_id`）。

## 7. 指标与结算口径（实现必须一致）

### 7.1 建议判定
- `implied_prob = 1 / odds_draw_place`
- `edge = base_prob - implied_prob`
- 缺硬门槛 -> `SKIP + missing_data`
- 不在窗口 `[2,6]h` -> `SKIP + out_of_window`
- `edge < 0.03` -> `SKIP + edge_low`
- `killswitch=STOP` -> `SKIP + killswitch_stop`

### 7.2 仓位
- `stake_pct = clamp((edge-0.03)/(0.10-0.03), 0, 1)`

### 7.3 paper bet 结算（stake=1）
- 命中：`pnl = (odds_draw_place - 1) * (1 - fee_rate) + rebate_rate`
- 未命中：`pnl = -(1 + fee_rate)`
- `roi = pnl`

### 7.4 Kill-Switch
- STOP：`roi_7d < -0.15` 或 `rolling_clv_50 < 0` 或 `max_drawdown > 0.30`
- WARN：`roi_7d < -0.08` 或 `max_drawdown > 0.20`

## 9. 验收标准
1. 建议生成可复现。
2. 决策透明可解释。
3. 结算闭环无漏单。
4. Kill-Switch 生效及时。
5. 管理端可用且可操作。
6. 代码质量达标（Controller 无业务、Vue 无直调、Adapter 集中清洗）。

### 9.1 测试计划矩阵（新增）
| 类型 | 覆盖点 | 通过标准 |
|---|---|---|
| 单元测试 | Edge/Stake/Decision 计算 | 覆盖率 >= 85% |
| 单元测试 | PnL/ROI 结算（win/loss/void） | 全通过 |
| 集成测试 | 采集->建议->下单->结算链路 | 全通过 |
| 集成测试 | Kill-Switch 手动与自动切换 | 状态准确 |
| E2E | 前端建议中心与监控页操作 | 无阻塞错误 |
| 回归测试 | 现有 AI/Poisson 功能不回退 | 全通过 |

### 9.2 LLM 能力验收标准（新增）
1. LLM 服务异常时，业务主流程不受影响（建议生成、风控判定、结算可正常运行）。
2. `AI解读` 文案生成耗时 P95 < 3s（可按环境调整）。
3. 对相同输入（同 `suggestion_id + prompt_version`）返回结果可复现或可追溯。
4. 所有 LLM 调用可审计：保存 `provider/model/prompt_version/request_id`。
5. 文案内容不出现硬编码投资承诺或“保证收益”表达。

  - 采集成功率 < 90%（10 分钟）告警
  - 结算延迟 > 120 分钟告警
  - STOP 状态触发立即告警

### 10.2 主要风险
1. 赛果回写依赖外部源（已设计降级）。
2. 赔率时序噪音（已限制窗口）。
3. ID 映射准确率（需人工抽检）。
4. 外部源不稳定（见第 13 节 SLA）。

### 10.3 LLM 集成风险与控制（新增）
1. **幻觉风险**：LLM 可能生成不准确结论。
   - 控制：LLM 仅做文案解释，不参与核心决策计算。
2. **延迟风险**：供应商抖动影响页面体验。
   - 控制：超时降级模板文案 + 异步生成 + 前端重试。
3. **成本风险**：高频调用带来费用上升。
   - 控制：同输入缓存、批量生成、按场景限频。
4. **合规风险**：文案出现误导性表述。
   - 控制：增加敏感词与合规规则过滤，禁止“保本/稳赚”等词。
5. **隐私风险**：将敏感数据送入第三方模型。
   - 控制：请求前脱敏，不传用户隐私、密钥、原始敏感日志。

## 12. 发布与回滚（新增）

### 12.1 发布步骤
1. 执行 DB 迁移。
2. 部署后端（关闭建议生成写入开关，先只读验证）。
3. 校验核心接口健康。
4. 部署前端。
5. 打开建议生成功能（灰度 10% -> 50% -> 100%）。

### 12.2 回滚策略
- 接口异常：回滚后端镜像版本。
- 数据异常：回滚开关到只读，保留数据不删除。
- 迁移异常：执行对应 downgrade（仅允许在未写入新业务数据时）。

## 13. 外部数据源 SLA 与降级（新增）

### 13.1 采集策略
- 采集频率：默认 5~10 分钟一次（赛前窗口内加密）。
- 超时：单请求 5 秒。
- 重试：指数退避最多 3 次。
- 限流：每源 QPS 控制，避免封禁。

### 13.2 降级策略
- 主源失败：切备用源。
- 双源失败：使用最近可信快照（标记 `stale_data=true`），并将建议强制降级为 `SKIP`（可配置）。
- 连续失败超过阈值：触发告警 + 可选自动切 `WARN`。

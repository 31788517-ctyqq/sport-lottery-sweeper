# 北单过滤决策系统完整整合方案（V2）

> 本文档在《北单过滤决策系统完整整合方案（V1）》基础上升级：将“规则过滤工具”演进为“可插拔决策管线 + 评分分层 + 组合约束求解 + 历史回测闭环”的策略决策系统。  
> **创建日期**：2026-03-04  
> **版本**：2.0  
> **状态**：草案（待评审）

## 0. V2 变更摘要（相对 V1）

### 0.1 为什么要做 V2
V1 已完成三维精算筛选与智能过滤模块整合方案，但仍存在：
- 过滤链路顺序固定，扩展成本高（新增过滤器需要改核心引擎）
- 只“筛掉/保留”，缺少“打分/分层/自动推荐胆码拖码”
- 胆拖组合存在组合爆炸风险，需要系统化剪枝/约束搜索
- 缺少历史回测闭环，策略质量无法量化对比与迭代
- 缺少可解释性与可观测性（每一步为什么过滤、耗时、漏斗变化）

### 0.2 V2 核心升级点（必须落地）
1. **Filter Pipeline（可插拔过滤管线）**：过滤器插件化 + Trace 追踪
2. **Score Engine（评分引擎）**：将场次分层为 A/B/C（胆候选/拖候选/备选）
3. **Combo Engine（组合引擎）**：约束搜索 + 早停剪枝 + 缓存复用
4. **Backtest Engine（回测引擎）**：同一套策略配置对历史逐期重放，输出 ROI/回撤/胜率
5. **Explain & Observability（解释与观测）**：每一步输出“为何通过/为何剔除/为何定胆”，并记录性能指标

## 1. 项目现状分析（继承 V1）

### 1.1 现有北单过滤系统概述
当前项目已实现基于三维精算筛选器的北单过滤系统，核心功能包括：

**1.1.1 三维筛选维度**
- **实力等级差 (ΔP)**: [-3, -2, -1, 0, +1, +2, +3]，反映主客队基础实力对比
- **赢盘等级差 (ΔWP)**: [-4, -3, -2, -1, 0, +1, +2, +3, +4]，反映盘路表现和"性格"特征
- **一赔稳定性 (P-Tier)**: [S(P1), A(P2), B(P3), B-(P4), C(P5), D(P6), E(P7)]，反映正路可信度

**1.1.2 核心业务规则**
- **降级规则**: 方向背离时P级强制下移一级
- **排序规则**: P3场次必须排在P4之前，同P级按ΔWP降序，P7按Ssum升序

### 1.2 技术架构现状
#### 后端架构
- **API层**: FastAPI (`/api/v1/beidan-filter/*`)
- **服务层**: `BeidanDataService`、`BeidanFilterEngine`
- **数据层**: PostgreSQL，`football_matches`表存储原始比赛数据
- **模型层**: `MatchModel`、`BeidanStrategy`

#### 前端架构
- **主组件**: `BeidanFilterPanel.vue`
- **状态管理**: Vue 3 Composition API / Pinia
- **UI框架**: Element Plus

### 1.3 已实现功能
1. 三维多选筛选
2. 实时匹配场次计数
3. 高级筛选与分页
4. 策略保存、加载、删除
5. 数据导出（CSV/JSON/Excel）
6. 移动端适配
7. 权限控制与操作日志

## 2. V2 总体架构升级：决策管线（Decision Pipeline）

### 2.1 逻辑架构（V2）
V2 在 V1 的“第四层智能过滤”基础上，进一步引入 **评分、组合、回测、解释、观测**，形成闭环：

```
原始比赛数据 / 赔率快照
    ↓
数据构建层（特征计算 + 缓存）
    ↓
Filter Pipeline（可插拔过滤器链 + Trace）
    ↓
Score Engine（打分 + A/B/C 分层 + 胆/拖候选）
    ↓
Combo Engine（组合生成 + 约束搜索 + 剪枝 + 缓存）
    ↓
Explain Engine（逐步解释：为何留下/剔除/定胆/剪枝）
    ↓
结果输出（排序 / 导出 / 投注单）
    ↓
Backtest Engine（历史重放 / ROI / 回撤 / 对比）
    ↓
策略迭代（版本管理 / A-B 对比 / 模板）
```

### 2.2 服务边界（推荐）
- `DecisionEngineService`：核心编排（pipeline → scoring → combo → explain）
- `BacktestService`：异步回测任务 + 报告生成
- `MetricsBuildService`：特征计算（可离线/定时/按需）

## 3. V2 数据模型与存储（支持赔率变化与回测）

> V1 以 `football_matches` 为主表。V2 建议采用“原始层 + 指标层 + 策略层 + 回测层”分层，便于重算与历史重放。

### 3.1 数据表建议（新增/扩展）

#### 3.1.1 原始数据层（Raw）
- `matches_raw`：比赛基础信息（联赛、队伍、时间、状态等）
- `odds_snapshots`：赔率快照（同一场可多时刻记录，用于赔率变化/资金方向判断）

#### 3.1.2 指标层（Metrics）
- `match_metrics`：三维字段 + 扩展特征（ΔP, ΔWP, P-Tier, odds_delta, form 等）
- `team_form_metrics`：球队近 N 场状态聚合（胜率、进球、失球、主客场差）

#### 3.1.3 策略与执行层（Strategy）
- `beidan_strategies`：策略定义（JSONB，含 pipeline/score/combo/backtest 配置）
- `beidan_strategy_versions`：策略版本（支持回滚与对比）
- `beidan_strategy_runs`：每次执行记录（输入 hash、过滤漏斗、耗时、输出数量）
- `beidan_combo_runs`：组合生成记录（组合数、剪枝率、约束命中统计）

#### 3.1.4 回测层（Backtest）
- `beidan_backtest_jobs`：回测任务（策略版本、时间区间、状态）
- `beidan_backtest_results`：逐期结果明细（命中/收益/投入）
- `beidan_backtest_metrics`：汇总指标（ROI、胜率、最大回撤、稳定性等）

### 3.2 Redis 缓存建议（强烈推荐）
- `feature:match_metrics:{match_id}:{metrics_version}`
- `preview:{hash(strategy_config+period)}`
- `combo:{hash(preview+constraints)}`
- `backtest:{job_id}:partial:{period}`

## 4. V2 过滤体系：Filter Pipeline（可插拔）

### 4.1 过滤器统一接口（含 Trace）
```python
class BaseFilter:
    name: str
    version: str

    def validate(self, config) -> None:
        ...

    def apply(self, matches, context, config):
        """return (matches, trace)"""
```

### 4.2 Pipeline 执行器
```python
class FilterPipeline:
    def __init__(self, filters):
        self.filters = filters

    def run(self, matches, context, configs):
        traces = []
        for f in self.filters:
            cfg = configs.get(f.name)
            f.validate(cfg)
            matches, trace = f.apply(matches, context, cfg)
            traces.append(trace)
        return matches, traces
```

### 4.3 默认过滤顺序（可在 UI 自定义）
1. `ThreeDimensionalFilter`（V1 现有）
2. `StatisticalTicketFilter`（对“整张方案”的统计约束，V2 中可前置做粗筛）
3. `PatternFilter`（连号/重号）
4. `CustomConditionFilter`（自定义条件）
5. `RiskControlFilter`（新增：热度/赔率剧烈波动/冲突降级）

> 说明：**“胆拖”不再作为过滤器**（避免逻辑混乱），转由 **Score Engine + Combo Engine** 负责。

## 5. V2 评分体系：Score Engine（胆/拖候选自动化）

### 5.1 输出结构
- `score_total`: 0~100
- `score_components`: 每项贡献（可解释）
- `tier`: A / B / C
  - A：胆候选（建议 1~3 场）
  - B：拖候选
  - C：备选/谨慎

### 5.2 示例加权（可配置）
- ΔP：30%
- ΔWP：25%
- P-Tier：25%
- 赔率变化（odds_delta / drift）：10%
- 近期状态（form）：10%

### 5.3 评分接口示例
```python
class ScoreEngine:
    def score(self, match, config) -> dict:
        # return {score_total, score_components, tier}
        ...
```

### 5.4 胆/拖推荐策略（默认）
- `tier=A` 且 `score_total >= A_threshold` → 胆候选
- `tier=B` 且 `score_total >= B_threshold` → 拖候选
- 允许用户手动覆盖（保留 V1 的标记逻辑作为“人工干预”）

## 6. V2 组合体系：Combo Engine（约束搜索 + 剪枝）

### 6.1 目标
在“候选胆/拖池”基础上生成投注组合，支持：
- 必含全部胆（或至少含 N 个胆）
- 票面最大场次（默认 8）
- 统计约束：胜负数量、赔率和值、奇偶比等（参考 V1 统计学映射）
- 模式约束：连号限制、历史重复相似度阈值

### 6.2 组合爆炸与剪枝
**约束搜索（Constraint Search）+ 早停剪枝（Early Pruning）**：
- 赔率和值超出范围 → 立即剪枝
- 连号超过上限 → 立即剪枝
- 当前组合理论最大可达分数不足 → 剪枝（可选优化）

### 6.3 组合接口示例
```python
class ComboEngine:
    def generate(self, dan_pool, tuo_pool, constraints, limits) -> dict:
        # return {combos, prune_stats, constraints_stats}
        ...
```

## 7. V2 可解释性：Explain Engine（必备）

### 7.1 解释内容
1. **过滤解释**：每个 Filter 的通过/剔除原因 + 数量变化漏斗
2. **评分解释**：score_components 与 tier 说明
3. **组合解释**：组合为何被保留/剪枝（命中哪条约束）

### 7.2 Trace 示例结构（建议）
```json
{
  "filter": "PatternFilter",
  "before": 58,
  "after": 36,
  "reasons_top": ["max_consecutive_same exceeded", "similarity>0.9"],
  "latency_ms": 12
}
```

## 8. V2 回测体系：Backtest Engine（策略闭环）

### 8.1 回测原则
- **同一份 StrategyConfig** 对历史逐期重放（确保线上与回测一致）
- 输出指标：
  - 胜率、命中率
  - ROI
  - 最大回撤（Max Drawdown）
  - 稳定性（波动/方差）
  - 按联赛/赔率区间/score 段贡献拆解

### 8.2 回测任务建议异步化
- 回测与大规模组合生成建议走任务队列（如 Celery/RQ/Arq）
- API 返回 `job_id`，前端轮询状态与报告

### 8.3 回测接口示例
```python
POST /api/v2/backtest/run
GET  /api/v2/backtest/{job_id}/status
GET  /api/v2/backtest/{job_id}/report
```

## 13. 风险控制（V2 补充）

| 风险项 | 影响程度 | V2 应对措施 |
|---|---:|---|
| 组合爆炸 | 高 | 约束搜索 + 早停剪枝 + 限制拖池规模 + 缓存 |
| 自定义条件安全 | 高 | AST 安全解释器/白名单字段/资源限制，禁用 eval |
| 线上与回测不一致 | 中 | StrategyConfig 单一事实来源，回测重放同一 pipeline |
| 性能瓶颈 | 中 | Redis 缓存、预计算指标、异步任务、数据库索引 |
| 可解释性缺失导致不信任 | 中 | Trace + 评分拆解 + 组合剪枝原因输出 |

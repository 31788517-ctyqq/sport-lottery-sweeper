### 0.1 术语边界（避免“策略”混淆）

本方案将“策略”明确拆为两层：
- **A 层：北单三维筛选器策略（基础策略）**
  - 即当前 `beidan_strategies` 里保存的筛选条件（三维条件 + 其他条件 + 排序）。
  - 作用：定义“先筛出哪些场次”。
- **B 层：决策系统策略（衍生策略）**
  - 在 A 层基础策略之上叠加 `scoring/combo/backtest/risk` 等决策配置。
  - 作用：定义“筛出后如何评分、组合、回测和执行”。

关系要求：
- B 层可引用 A 层策略作为基底（可覆盖部分参数），也可独立定义。
- A 层继续独立存在，不被 B 层替代。

## 2. 当前差距与关键问题（Gap）

1. **Pipeline 未插件化**  
当前是 `advanced-filter -> BeidanDataService.get_filtered_matches` 的单体流程，不是可插拔 Filter Pipeline。

2. **三维参数存在“名实不一致”**  
`threeDimensional` 字段多数为占位，实际生效主要依赖 `otherConditions.powerDiffs/winPanDiffs/stabilityTiers`。

3. **降级开关未真正生效**  
前端传 `includeDerating`，但服务层未落地对应降级逻辑。

4. **历史过滤引擎代码未接入主链**  
`backend/app/services/filter_engine.py` 存在，但未被主 API 调用。

5. **多策略存在策略语义错位**  
前端选择的是 A 层“已保存筛选策略”，后端调度器默认执行内置策略函数，未体现“B 层可基于 A 层或独立定义”的双模式链路。

6. **无组合引擎与回测引擎**  
当前没有 `combo/backtest` 生产级链路与接口。

7. **可观测性不足**  
虽有执行日志表，但缺少完整漏斗、延迟、剪枝率、分层命中等统一指标。

## 3. 修订后的 V2 目标（To-Be）

在不破坏现有前端和 API 的前提下，V2.1 目标如下：

1. 将现有高级筛选重构为可插拔 `Filter Pipeline`（保留现有接口）。
2. 增加 `Score Engine`（A/B/C 分层与可解释评分）并回填到筛选结果。
3. 新增 `Combo Engine`（组合生成、约束搜索、剪枝统计）。
4. 新增 `Backtest Engine`（同策略配置历史重放）。
5. 补齐 `Explain + Observability`（可解释 trace + 指标入库）。
6. 统一“多策略”语义：调度任务执行 B 层决策策略（可引用 A 层或独立定义），而非独立内置策略函数。
7. 管理后台统一入口：`精算工具中心 -> 北单过滤软件`（Web/PC）。

## 4. 架构修订方案（基于现有代码增量）

### 4.1 API 策略（保持兼容）

保留现有接口不变：
- `/api/v1/beidan-filter/advanced-filter`
- `/api/v1/beidan-filter/strategies*`

在同一命名空间增量新增：

- `GET /api/v1/beidan-filter/backtest/{job_id}/status`
- `GET /api/v1/beidan-filter/backtest/{job_id}/report`

说明：先不单独切 `/api/v2`，避免现网前端与脚本迁移成本。

### 4.2 服务拆分（落地路径）

在 `backend/services/` 下新增并逐步替换：
- `decision_pipeline_service.py`：编排 filter -> score -> explain
- `filters/`：`base_filter.py`, `three_dimensional_filter.py`, `line_id_range_filter.py`, `risk_control_filter.py`
- `score_engine.py`：输出 `score_total/score_components/tier`
- `combo_engine.py`：约束搜索 + 剪枝统计
- `backtest_service.py`：异步任务与报告

现有 `BeidanDataService` 继续保留，逐步下沉为“数据装载与字段标准化层”。

### 4.3 配置模型（统一 StrategyConfig）

采用“分层配置”：
- A 层（基础策略）：保留 `beidan_strategies` 现有结构和语义，不强改。
- B 层（决策策略）：新增统一 `DecisionStrategyConfig`，核心字段含：
  - `base_filter_strategy_ref`（可选，引用 A 层策略 id/name）
  - `pipeline/scoring/combo/backtest/output`
  - `overrides`（可选，用于覆盖 A 层部分筛选项；当未引用 A 层时可省略）

迁移策略：
- A 层：保持原样，继续被北单三维筛选器直接使用。
- B 层：单独建模与存储，避免污染 A 层数据结构。

### 4.4 数据表修订（最小增量）

保留现有：
- `beidan_strategies`（A 层基础策略）
- `beidan_strategy_execution_logs`（A 层筛选执行日志）

新增建议：
- `beidan_decision_strategies`（B 层决策策略，`base_filter_strategy_id` 可空）
- `beidan_decision_strategy_versions`（B 层版本快照）
- `beidan_combo_runs`（B 层组合执行与剪枝统计）
- `beidan_backtest_jobs`（B 层回测任务）
- `beidan_backtest_results`（B 层逐期结果）
- `beidan_backtest_metrics`（B 层汇总指标）

可选增强：
- 若 `odds_snapshots` 仍为空，先补采集任务再启用赔率漂移特征。

### 4.5 后台菜单与产品入口（本轮必做）

## 5. 多策略模块修订（必须优先）

### 5.1 问题定义
当前多策略执行器以“内置策略函数”运行，而业务要求是：执行 B 层决策策略，且 B 层既可引用 A 层基础策略，也可独立运行。

### 5.2 修订方案
- 调度器执行逻辑改为：按 B 层决策策略执行；B 层配置可选引用 A 层策略。
- 当仅选择 A 层策略时，系统可自动生成临时 B 层（默认 scoring/combo 配置）执行。
- 保留内置策略仅作调试用途，不作为默认生产链路。
- `GET /api/v1/multi-strategy/strategies` 返回“可执行策略实例”（A 层 + B 层），而非硬编码列表。

### 5.3 验收标准
- 手动执行与定时执行使用同一 B 层配置、同一输出结构。
- 钉钉推送场次数与前端同策略筛选结果可对齐（允许时间窗口差异）。
- 可追溯到“该次执行对应哪个 B 层决策版本，以及是否引用了 A 层基础策略”。

- 抽离过滤器接口与执行器。

- 增加 trace（before/after/latency/reasons）。

交付：`advanced-filter` 内部实现切换为 pipeline，接口不变。

### Phase 2（1~1.5 周）评分分层
- 落地 `ScoreEngine`。

交付：决策预览与分层推荐可用。

### Phase 3（1.5~2 周）组合引擎
- 新增 `decision/combo` 接口。
- 支持胆拖池、约束、剪枝统计。
- 输出组合数量、命中约束、剪枝原因。

交付：可用组合生成 MVP。

### Phase 4（2 周）回测闭环

### Phase 5（持续）观测与优化
- 指标入库：过滤漏斗、阶段耗时、剪枝率、回测性能。
- 数据质量告警：空赔率快照、策略配置异常、结果漂移。

## 8. 风险与控制

| 风险 | 影响 | 控制措施 |
|---|---:|---|
| 组合爆炸 | 高 | 约束搜索、早停剪枝、拖池上限、缓存 |
| A/B 两层策略混淆 | 高 | 强制分层命名、独立存储、明确 B 层可选引用 A 层 |
| 多策略结果不一致 | 高 | 统一改为执行 B 层配置（可选引用 A 层） |
| 线上/回测不一致 | 高 | 同一 StrategyConfig、同一 pipeline |
| 赔率特征不可用 | 中 | 先补 `odds_snapshots` 数据采集，再启用相关特征 |
| 性能劣化 | 中 | 分阶段压测、缓存、异步化回测 |

### 10.1 IDE 协同落地原则（执行约束）

- 仅维护本文件：V2.1-DEV 拆解直接写入当前文档，不新建并行版本文件。

# 后台用户管理与 RBAC 改造可行性评估与优化方案（v1.1）

> 目的：在正式改造前，先确认当前文档是否可落地，并输出一份可直接开发执行的优化版方案。  
> 范围：仅文档评估与优化，不涉及代码修改。

## 1. 评估结论（先给结论）

### 1.1 总体可行性
- 结论：`可行，但必须分阶段实施`。
- 原因：项目已有用户/角色/权限/部门基础接口与页面，但存在接口前缀不统一、鉴权链路分散、历史兼容路由较多的问题，直接一次性上“动态路由 + 数据权限 + 审批流”风险高。

### 1.2 立即可做（P0/P1）
- 统一接口契约与前缀（`/api/v1/admin/*` 为主）。
- 统一前端权限入口（路由守卫 + 指令 + store）。
- JWT 即时失效（`token_version`）与会话治理。
- 审计日志关键动作补齐（改权、重置密码、禁用/启用、删除用户）。

### 1.3 需要二期推进（P2/P3）
- 服务端动态路由（Server-Driven Routing）。
- 数据权限（data_scope）全链路注入。
- 四眼原则审批流、异常行为分析大屏等增强能力。

## 2. 当前现状与约束（基于仓库可见结构）

### 2.1 已具备能力
- 后端已有用户管理主接口：`/api/v1/admin/admin-users/*`。
- 后端已有角色/权限/部门接口：`/api/v1/admin/roles`、`/permissions`、`/departments`。
- 前端已有用户列表、角色权限、部门管理页面与基础权限工具（store/directive/composables）。

### 2.2 明确问题
- 接口路径混用：`/api/v1/admin/*` 与 `/api/admin/*` 并存（前端模块中仍有历史路径）。
- 权限入口分散：不同模块各自判断角色/权限，策略不一致。
- 用户管理路由并存过多：`admin-users`、`users`、`frontend-users`、`simple-users` 同时存在，职责边界不清。
- 文档与部分文案存在编码问题，影响维护与交付质量。

### 2.3 实施约束
- 需要保持现网可用，避免一次性替换导致登录、菜单、权限全链路回归风险。
- 需兼容已有前端页面，采取“兼容层 + 逐步切换”。
- 鉴权与权限改造属于高风险改动，必须有灰度开关与回滚策略。

## 3. 原建议项可行性评级（优化后）

| 建议项 | 可行性 | 推荐阶段 | 说明 |
|---|---|---|---|
| JWT 即时失效（token_version） | 高 | P1 | 改造范围可控，安全收益高 |
| 前后端枚举同步（统一字典） | 高 | P1 | 改造成本低，能降低联调错误 |
| 指令化按钮权限（`v-auth`） | 高 | P1 | 提升前端一致性，易落地 |
| 审计日志闭环 | 高 | P1 | 合规与追踪基础能力，必做 |
| 数据权限（data_scope） | 中 | P2 | 需改模型与查询层，建议分模块落地 |
| 服务端动态路由 | 中低 | P3 | 牵涉前后端架构，建议作为 V2 |
| 敏感字段脱敏 + 明文查看审计 | 中 | P2 | 需明确权限与场景边界 |
| 四眼原则审批流 | 中 | P3 | 业务流程改造较大 |
| 审计可视化分析 | 中 | P3 | 依赖前序日志标准化完成 |

## 4. 优化后的目标与边界

### 4.1 V1 目标（本轮必须达成）
- 用户管理主链路稳定：登录、列表、新增、编辑、状态切换、角色分配、重置密码可用。
- 权限判定统一：页面级、按钮级逻辑由统一入口提供。
- 安全基线到位：Token 可即时失效，关键动作有审计记录。
- 接口契约收敛：前端用户管理改造仅消费一套主接口前缀。

### 4.2 V1 明确不做
- 不在本轮做全面服务端动态路由。
- 不在本轮做完整审批工作流引擎。
- 不在本轮做全量业务表 data_scope 改造（仅预留能力和样例）。

## 5. 分阶段落地清单（可直接执行）

## 5.1 P0 基线收敛（1-2 天）
- 输出“用户管理接口主契约表（唯一真相）”：
  - 用户：`/api/v1/admin/admin-users/*`
  - 角色：`/api/v1/admin/roles/*`
  - 权限：`/api/v1/admin/permissions/*`
  - 部门：`/api/v1/admin/departments/*`
- 标记所有历史路径调用点与兼容策略（保留/重定向/下线时间）。
- 统一错误码与响应包装约定（`code/data/message`）。

交付物：
- 《RBAC 契约基线表 v1.1》
- 《历史接口兼容清单与下线计划》

## 5.2 P1 核心改造（3-5 天）
- 鉴权：
  - 实施 `token_version` 机制。
  - 权限变更、禁用用户、重置密码后触发旧 token 失效。
- 前端权限：
  - 统一路由守卫权限入口。
  - 统一按钮权限指令用法，清理重复判断。
- 审计：
  - 关键动作强制审计字段标准化（操作者、对象、动作、前后值、IP、UA、结果）。

交付物：
- 《鉴权与会话治理说明》
- 《权限入口统一规范》
- 《审计日志字段规范 v1》

## 5.3 P2 增强改造（5-8 天）
- 数据权限（先做基础型）：
  - 角色增加 `data_scope`（ALL/DEPT/SELF/CUSTOM）。
  - 在用户与部门查询链路先落地过滤注入。
- 敏感字段策略：
  - 列表默认脱敏；
  - 明文查看需单独权限并记录审计。

交付物：
- 《data_scope 设计与首批落地范围》
- 《敏感信息查看与审计规范》

## 5.4 P3 二期架构（按版本排期）
- 服务端动态路由。
- 四眼原则审批流（关键配置变更）。
- 审计数据可视化与异常检测。

交付物：
- 《RBAC V2 架构设计稿》

## 6. 风险与回滚策略

### 6.1 高风险点
- 路径切换导致前端 404/405。
- token 失效策略上线后导致“频繁掉线”。
- 权限收敛后暴露历史越权调用。

### 6.2 控制措施
- 灰度开关：
  - `RBAC_ENFORCE_STRICT`
  - `RBAC_TOKEN_VERSION_ENABLED`
  - `RBAC_DATA_SCOPE_ENABLED`
- 双轨兼容期：
  - 主接口切换后保留旧接口只读兼容（明确结束日期）。
- 回滚：
  - 关闭严格策略开关；
  - 回切旧鉴权判定；
  - 保留审计链路不断流。

## 7. 验收标准（改造前先对齐）

### 7.1 功能验收
- 管理员与普通角色在菜单、页面、按钮三个层级权限表现一致。
- 用户状态切换、角色分配、重置密码接口无 4xx/5xx 异常回归。

### 7.2 安全验收
- 修改角色权限后，旧 token 在可接受时间内失效（目标：秒级到分钟级）。
- 关键动作审计记录完整且可检索。

### 7.3 工程验收
- 用户管理相关前端 API 不再混用 `/api/admin/*` 与 `/api/v1/admin/*`。
- 联调文档、错误码、字段定义一致且可追踪版本。

## 8. 建议的实施顺序（推荐）
1. P0：先收敛契约与路径，冻结“唯一真相”。
2. P1：再做鉴权与权限入口统一，确保主链路稳定。
3. P2：补数据权限与脱敏能力。
4. P3：最后推进动态路由和审批流等架构增强。

# 从当前多分支收敛到单 `main` 的无代码操作清单（低风险顺序）

## 1. 目标与原则

### 1.1 目标
- 仓库长期仅保留一个常驻分支：`main`。
- 历史发布与里程碑通过 `Tag` 管理，不再依赖常驻发布分支。
- 合并策略统一为：变更先在临时分支完成，合并后立即删除临时分支。

### 1.2 原则
- 先“可回退”，再“做切换”，最后“删分支”。
- 不做代码改动，只做仓库治理动作。
- 每一步都有可验证结果，未通过不进入下一步。

---

## 2. 执行前检查（T-1 天）

### 2.1 人员与窗口
- 确定执行窗口（建议低峰期 30~60 分钟）。
- 通知所有开发者在窗口内暂停向 `main/master/develop/stable-base/backup/*` 推送。
- 指定 1 名执行人 + 1 名复核人。

### 2.2 仓库健康
- GitHub Actions 最近一次关键流程（CI、E2E、部署前检查）为绿色。
- 当前线上发布版本对应提交可定位（commit/tag 可查）。
- 分支保护规则当前配置已截图留档。

### 2.3 分支清点
- 导出当前远端分支清单（含最后提交时间、提交人）。
- 标记“仍在开发中”的分支与负责人。
- 标记“可归档后删除”的历史分支。

**通过标准**
- 分支列表完整，且每个分支有明确归类（保留/待合并/待归档/待删除）。

---

## 3. 备份与回退锚点（T-0 开始）

### 3.1 创建保护性 Tag（推荐）
- 在以下关键分支最新提交打 `Tag`：
  - `main`（现主线）
  - `master`（若存在）
  - `stable-base`（若存在）
  - `develop`（若存在）
- Tag 命名建议：
  - `pre-branch-consolidation-main-YYYYMMDD`
  - `pre-branch-consolidation-master-YYYYMMDD`

### 3.2 生成分支快照清单
- 记录每个将被删除分支的：
  - 分支名
  - 最后提交 SHA
  - 最后提交时间
  - 负责人

**通过标准**
- 任一即将删除分支都能通过 Tag 或 SHA 找回。

---

## 4. 冻结与变更管控（T-0）

### 4.1 临时冻结合并入口
- 暂时提高 `main` 保护强度（仅管理员可合并）。
- 暂停自动合并（Auto-merge）功能。
- 暂停机器人批量提交（若有定时 bot）。

### 4.2 清理“未完成 PR”
- 对非 `main` 目标分支的 PR 做处理：
  - 可转目标到 `main` 的直接改目标。
  - 暂不处理的标记 `post-consolidation` 标签。

**通过标准**
- 所有活跃 PR 的目标分支均为 `main` 或已明确延期。

---

## 5. 主线统一（核心步骤）

### 5.1 确认默认分支
- GitHub Settings -> Default branch 设为 `main`。
- 确认仓库首页默认进入 `main`。

### 5.2 保护规则统一到 `main`
- 将原 `master/stable-base/develop` 的关键保护项迁移到 `main`：
  - Require PR
  - Required checks
  - Required approvals
  - Restrict force push / delete
  - Require up-to-date branch before merge

### 5.3 CI 触发分支统一
- 检查 workflow 触发条件中的常驻分支，仅保留 `main`（必要时保留临时窗口分支 1~2 天）。
- 部署流程仅接受来自 `main` 的发布动作。

**通过标准**
- 默认分支是 `main`。
- `main` 保护规则与既定治理规则一致。
- 新 PR 默认指向 `main`，关键 workflow 正常触发。

---

## 6. 分支收敛（按风险顺序）

### 6.1 先收敛低风险历史分支
- 删除长期无活动且已归档的 `backup/*`、`test/*`、`temp/*` 分支。

### 6.2 再收敛中风险常用分支
- `develop`、`stable-base`、`master`（若存在）按以下顺序处理：
  1. 对比与 `main` 差异（提交数、文件数、最近提交时间）。
  2. 若存在未入 `main` 的必要变更，先通过 PR 合入 `main`。
  3. 合并后再次确认差异为 0，再删除分支。

### 6.3 最后处理高风险“有人在用”的分支
- 与负责人确认“已迁移到新临时分支并完成 PR”后删除。

**通过标准**
- 远端仅剩常驻分支 `main`（以及短期临时分支，若保留需有删除日期）。

---

## 7. 本地与协同工具同步

### 7.1 开发者本地同步
- 所有人本地默认分支切到 `main`。
- 清理本地已删除远端分支的跟踪关系。

### 7.2 IDE / 插件配置同步
- 更新各 IDE 的默认 PR 目标分支为 `main`。
- 更新自动提交/自动创建分支模板（前缀可保留：`feature/*`、`fix/*`）。
- 更新文档与模板（PR 模板、协作规范文档）。

**通过标准**
- 新建分支与新建 PR 的默认目标均为 `main`。

---

## 8. 验收清单（收尾）

### 8.1 仓库侧
- Default branch = `main`。
- Branch protection 仅对 `main` 生效（或额外临时规则有明确下线时间）。
- 关键 CI required checks 绑定到 `main`。

### 8.2 流程侧
- 新 PR 可正常创建并合并到 `main`。
- 合并后可自动删除临时分支。
- 发布流程从 `main` 正常触发一次并通过。

### 8.3 文档侧
- 多 IDE 协同文档、发布文档、分支治理文档均更新为单 `main` 口径。

---

## 9. 回退预案（必须保留）

若收敛后 24 小时内出现流程阻塞：
- 使用 `pre-branch-consolidation-*` Tag 快速恢复对应分支。
- 暂时恢复必要分支保护规则（只恢复最小集合）。
- 记录回退原因并在 48 小时内完成二次收敛评审。

---

## 10. 推荐时间线（示例）

- D-1：分支盘点、通知冻结、准备 Tag 策略。
- D-Day：
  1. 打 Tag 与快照归档
  2. 切默认分支与保护规则
  3. 合并必要差异
  4. 删除冗余分支
  5. 执行一次完整 CI + 发布冒烟
- D+1：观察与补漏，确认无回退后关闭迁移任务。

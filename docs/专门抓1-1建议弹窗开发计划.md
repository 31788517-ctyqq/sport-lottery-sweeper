# 专门抓1-1“建议”弹窗开发计划（详细版）

## 0. 文档定位
- 适用页面：`/admin/draw-prediction/poisson-11`
- 功能入口：比赛列表“操作”列新增“建议”按钮，打开建议弹窗。
- 目标：把模型结果、价值判断、风险约束整合为可执行投注建议。

## 1. 依据文档
本计划基于以下三份文档统一口径：
1. `docs/专抓1-1开发计划.md`（总体模块目标与里程碑）
2. `docs/专门抓1-1赚钱思路.md`（建议弹窗业务框架与文案口径）
3. `docs/专门抓1-1建议弹窗字段映射.md`（字段映射与就绪度）

## 2. 建设目标与原则

### 2.1 建设目标
1. 在单场维度输出明确结论：`强建议/可跟进/仅观察/放弃`。
2. 保证建议“可解释”：每个结论必须可追溯到字段与规则。
3. 保证建议“可执行”：包含仓位与风控，不只给概率。
4. 保证建议“可复盘”：沉淀建议与赛果，用于后续迭代。

### 2.2 设计原则
1. 先结论，后细节：第一屏必须看到“是否下注”。
2. 无数据不猜测：缺失字段统一显示 `-`，并降低建议等级。
3. 可迭代分层：MVP用现有字段；扩展项独立接入，不破坏结构。
4. 风险优先：任何触发“一票否决”直接放弃。

## 3. 范围定义

### 3.1 本期范围（In Scope）
1. `poisson-11` 列表行级新增“建议”按钮与弹窗。
2. 弹窗六大分区：结论卡、模型面、价值面、结构面、风险面、资金面。
3. 基于现有 `poisson-11/detail` 字段完成MVP建议计算。
4. 固定文案、缺失兜底、建议分级、风险否决规则。
5. 复盘字段结构设计（可先存前端草稿结构，后续落库）。

### 3.2 非本期范围（Out of Scope）
1. 新增复杂AI解释模型。
2. 阵容/战意/赛程风险的完整自动化抓取与建模。
3. 资金管理策略的个性化账户联动。

## 4. 当前数据现状（可直接复用）

### 4.1 可直接使用接口
1. `GET /api/v1/draw-prediction/poisson-11/list`
2. `GET /api/v1/draw-prediction/poisson-11/detail/{match_id}`

### 4.2 当前已就绪关键字段
1. 概率与排名：`prob_11`、`rank`
2. 模型中间量：`mu_total`、`mu_diff`、`mu_home`、`mu_away`
3. 盘口核心：`input_payload.total_goals_line`、`input_payload.handicap`
4. 平赔与风险辅助：`input_payload.odds.draw`、`input_payload.quality_flags[]`
5. 比分/状态提示：`source_attributes.full_score/halftime_score/status_des`

### 4.3 当前缺口字段
1. `1-1比分赔率 Odds_11`（来源已明确为 500 比分玩法，待抓取入库）
2. 阵容风险（结构化缺失）
3. 战意风险（结构化缺失）
4. 赛程风险（结构化缺失）

### 4.4 500比分赔率来源确认（已验证）
1. 主页面：`https://trade.500.com/bjdc/project_fq_bf.php?expect={period}`
2. 期号参数：`expect`（页面下拉 `#expect_select` 与隐藏字段 `#expect` 同步）
3. 场次结构：`tr.vs_lines` 提供 `index/scheduleDate/endTime/homeTeam/guestTeam`，对应展开行 `tr#sp_lines_{index}`
4. 赔率定位：在 `sp_lines_{index}` 内定位 `span.f_bf` 文本为 `1:1` 的 label，读取 `span.sp_value`
5. 回退源：`https://www.500.com/static/public/bjdc/xml/sp/just_{period}_42.xml`
6. 字段映射：`playid=42`（比分）下，`1:1` 对应第13项，即 XML `c13`

## 5. 弹窗信息架构（页面结构）

### 5.1 结论卡（首屏）
- 字段：建议等级、操作建议、建议仓位、核心理由（最多3条）。
- 交互：固定置顶；首屏可见。

### 5.2 模型面
- 字段：`P(1-1)`、`rank`、`μT`、`μD`、`μH`、`μA`、模型来源完整度。
- 说明：用于回答“模型是否支持下注”。

### 5.3 价值面
- 字段：`Odds_11`、`P_imp`、`P_model`、`Edge`、`Margin`、价值结论。
- 现状：`Odds_11` 数据源已确认；抓取任务上线前整块显示“待接入/ - ”，上线后启用价值计算。

### 5.4 结构面
- 字段：大小球区间、亚洲让球区间、平赔区间、结构标签。
- 说明：映射“1-1甜点区”逻辑。

### 5.5 风险面
- 字段：数据缺失风险、盘口异常风险、状态/比分风险提示、阵容/战意/赛程（预留）。
- 交互：触发高风险时显示红色告警标签。

### 5.6 资金面
- 字段：单场上限、当日上限、连亏保护、执行纪律提示。
- 说明：用于抑制情绪下单。

## 6. 建议计算规则（MVP版）

### 6.1 综合评分（100分）
1. 模型强度：40分
2. 价值优势：30分
3. 结构匹配：20分
4. 风险控制：10分

### 6.2 分级阈值
1. `80-100`：强建议
2. `65-79`：可跟进
3. `50-64`：仅观察
4. `<50`：放弃

### 6.3 一票否决
满足任一条件直接“放弃”：
1. 主抓 + 回退后仍缺失 `Odds_11` 且无法计算价值项（MVP默认触发降级，若策略要求可直接否决）。
2. `Edge <= 0`（当有Odds_11时启用）。
3. 关键字段缺失且 `quality_flags` 显示默认值过多。
4. 状态异常（如已完场仍尝试建议下注）。

### 6.4 无数据兜底规则
1. 任意缺失字段显示 `-`。
2. 依赖计算字段若上游缺失，则结果显示 `-`。
3. 核心字段不足时，结论文案固定为：`数据不足，建议观望`。

## 7. 字段映射执行口径（实现约束）

### 7.1 映射优先级
1. 详情顶层字段优先（稳定、直接）。
2. `input_payload` 其次（模型计算上下文）。
3. `source_attributes` 兜底（源数据透传）。

### 7.2 关键映射示例
1. `P_model` ← `detail.prob_11`
2. `μT/μD/μH/μA` ← `detail.mu_*`（若空则回退 `input_payload.mu.*`）
3. `大小球区间` ← `input_payload.total_goals_line`
4. `让球区间` ← `input_payload.handicap`
5. `平赔区间` ← `input_payload.odds.draw`
6. `风险缺失` ← `input_payload.quality_flags[]`
7. `状态提示` ← `source_attributes.status_des/full_score/halftime_score`

## 8. 前后端实施方案

### 8.1 前端任务
1. 操作列新增“建议”按钮。
2. 新增建议弹窗组件（或在 `Poisson11Scanner.vue` 内实现）。
3. 弹窗分区渲染、字段映射、缺失兜底、评分计算。
4. 建议等级样式、风险告警样式、待接入标记。
5. 与“模型数据”弹窗并行，不相互覆盖。

### 8.2 后端任务（本期最小化）
1. 复用现有 `detail` 接口，不强制新增接口。
2. 确保 `input_payload` 关键字段稳定返回。
3. 可选：新增“建议解释字段”接口（下一阶段）。

### 8.3 数据任务（比分赔率优先）
1. 主抓入口：`project_fq_bf.php?expect={period}`，按期号拉取页面并解析比分 SP。
2. 主抓解析：逐场读取 `vs_lines.index`，进入 `sp_lines_{index}`，提取 `f_bf=1:1` 对应 `sp_value`。
3. 回退解析：主抓缺失时请求 `just_{period}_42.xml`，读取 `w{index}@c13`。
4. 关联规则：优先 `period + number(index/chnum)`；回退 `schedule_date + home_team + away_team + kickoff(±5m)`。
5. 落库字段：`source_attributes.odds_score_11`、`odds_score_11_source`、`odds_score_11_period`、`odds_score_11_updated_at`。
6. 缺失口径：值为 `--/空/0/非数值` 统一记为缺失并前端显示 `-`。
7. 后续继续补齐阵容/战意/赛程风险结构化字段。

## 10. 测试计划

### 10.1 功能测试
1. 按钮是否正确打开对应场次弹窗。
2. 每个分区字段是否按映射规则显示。
3. 缺失字段是否统一显示 `-`。
4. 不同评分区间是否正确映射建议等级。

### 10.2 异常测试
1. 接口超时/失败时弹窗提示是否友好。
2. `input_payload` 缺字段时是否不报错。
3. 已完场比赛是否触发风险提示。

### 10.3 回归测试
1. 不影响现有“模型数据”弹窗。
2. 不影响列表分页、查询、期号筛选。

## 11. 验收标准（DoD）
1. `poisson-11` 每场比赛可打开“建议”弹窗。
2. 弹窗六大分区可完整渲染，字段有据可依。
3. 缺失字段全部按 `-` 处理，无异常崩溃。
4. 建议等级、操作建议、仓位建议可稳定输出。
5. `Odds_11` 能按期号抓取并正确映射到比赛（含 `1:1` 值）。
6. 至少完成 20 场样本人工验收，结论与规则一致。

## 12. 风险与应对
1. 500 页面结构变更：保留 XML 回退抓取，页面解析规则做可配置化。
2. 字段波动：通过映射优先级与兜底减少页面失败。
3. 规则争议：把评分权重配置化，支持快速调参。
4. 误用风险：弹窗显著展示“非收益承诺”与仓位上限。

## 13. 复盘与迭代
建议落地后，每场沉淀以下复盘字段：
1. 赛前建议等级
2. 是否实际投注
3. 投注仓位
4. 核心理由
5. 实际赛果（是否1-1）
6. 复盘结论（命中/偏差原因）

> 迭代目标：由“静态建议弹窗”升级为“可学习、可调参、可复盘”的决策组件。

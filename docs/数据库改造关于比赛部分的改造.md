数据库改造关于比赛部分的改造

一、为了整合不同来源足球球数据并构建强健的数据库参考：
为了整合不同来源的足球数据并构建健壮的数据库，我建议你采用标准化与灵活性并重的设计思路。

核心思想是：创建一个统一、标准化的核心表（matches），再为每个数据源建立独立的、可灵活扩展的“扩展表”或“属性表”。
下图清晰地展示了这种设计的核心架构：
第一步：设计核心比赛表 (matches)
这是所有数据来源的统一入口，存储最通用、最确定的信息。
sql
CREATE TABLE matches (
    match_id BIGINT PRIMARY KEY AUTO_INCREMENT, -- 自增主键，内部使用
    source_match_id VARCHAR(255) NOT NULL,      -- 数据源提供的原始ID
    data_source VARCHAR(50) NOT NULL,           -- 数据源名称，如 'source_100qiu'
    match_time DATETIME NOT NULL,               -- 标准化后的比赛时间
    league VARCHAR(100),                        -- 联赛名称
    league_short_name VARCHAR(50),              -- 联赛简称
    home_team VARCHAR(100) NOT NULL,
    away_team VARCHAR(100) NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    UNIQUE KEY unique_source_match (data_source, source_match_id) -- 防止同一源数据重复);
关键点：source_match_id 和 data_source 组成唯一约束，确保同一数据源的同一场比赛只存一次。
第二步：设计扩展表 (source_xxx_attributes)
为每个数据源创建独立的扩展表，用于存储其特有的、命名不规则的字段。以你提供的“100qiu”为例：
sql
CREATE TABLE source_100qiu_attributes (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    match_id BIGINT NOT NULL, -- 关联核心表
    rq VARCHAR(10),           -- 让球，如 '1', '-1'
    home_power INT,
    guest_power INT,
    home_win_pan FLOAT,
    guest_win_pan FLOAT,
    home_feature VARCHAR(255),
    guest_feature VARCHAR(255),
    home_enter_efficiency VARCHAR(50),
    home_prevent_efficiency VARCHAR(50),
    -- ... 其他所有特有字段
    jiaoFenDesc TEXT,
    jiaoFenMatch1 TEXT,
    jiaoFenMatch2 TEXT,
    FOREIGN KEY (match_id) REFERENCES matches(match_id) ON DELETE CASCADE);
设计优势：

灵活映射：字段名可与数据源API完全一致，简化ETL（数据抽取、转换、加载）过程。


隔离变化：某个数据源的字段变动或增减，不影响其他数据源和核心表结构。


查询清晰：通过 JOIN 关联查询，数据归属一目了然。

第三步：建立通用属性表 (unified_attributes) (可选但推荐)
将最重要、最通用的分析指标从各扩展表中提取出来，进行标准化命名和统一存储。这是进行跨数据源对比分析的关键。
sql
CREATE TABLE unified_attributes (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    match_id BIGINT NOT NULL,
    attribute_type VARCHAR(50) NOT NULL, -- 指标类型，如 'asian_handicap', 'pre_win_odds'
    attribute_name VARCHAR(100) NOT NULL, -- 标准化指标名，如 'pre_handicap', 'home_win_odds'
    string_value VARCHAR(255),
    decimal_value DECIMAL(12, 4),
    int_value INT,
    source_origin VARCHAR(255), -- 记录原始字段名和来源，如 'source_100qiu.rq'
    FOREIGN KEY (match_id) REFERENCES matches(match_id) ON DELETE CASCADE,
    INDEX idx_type_name (attribute_type, attribute_name));
这个表的使用场景示例：
假设数据源A的让球叫 rq，数据源B的叫 handicap_line，你可以将它们统一转换为标准名 pre_handicap 存入此表。
sql

编写统一的转换函数：为每个数据源编写专门的函数，将其原始数据转换为核心表(matches)和对应扩展表的格式。这是ETL过程的核心。

配置与日志：将数据源的API地址、字段映射关系写入配置文件。记录详细的数据拉取和转换日志，便于排查数据不一致或缺失的问题。

总结：最佳实践路径

从简开始：如果数据源不多，可以先实现核心表 matches + 各数据源扩展表的方案，快速上线。


逐步统一：随着分析需求增加，再引入 unified_attributes 通用属性表，将核心指标逐步标准化。


保持映射：始终坚持程序中的模型字段名与数据库字段名保持一致，这是避免混淆和错误的最简单法则。


预留扩展：在核心表和扩展表中预留 json_data 或 extra_fields 字段，用于存储无法预知的新字段。

这个设计在规范性和灵活性之间取得了很好的平衡，既能保证核心数据整洁，又能轻松应对不同数据源的千差万别，是处理多源异构数据的经典模式。


二、核心比赛表设计的设计参考：
基于足球竞猜的深度分析需求，核心比赛表设计应该遵循“层次化、可追溯、易扩展”的原则。我来为你提供具体的设计建议：
🏗️ 核心表结构设计（三层次模型）
第一层：比赛基础信息表 (match_basic)
存储不可变的比赛标识和基本信息
sql
CREATE TABLE match_basic (
    -- 主键与标识
    match_id CHAR(32) PRIMARY KEY,  -- 使用UUID或MD5(来源+源ID)生成全局唯一ID
    internal_match_no VARCHAR(50) UNIQUE,  -- 内部比赛编号: 如"20241231_001_AF"
    
    -- 数据源标识（关键设计）
    data_source VARCHAR(50) NOT NULL,      -- 数据源系统编码
    source_match_id VARCHAR(255) NOT NULL,  -- 数据源的原始ID
    data_version INT DEFAULT 1,             -- 数据版本号，用于追踪变更
    
    -- 比赛基础信息
    match_time DATETIME NOT NULL,           -- 开赛时间（UTC时间）
    match_time_local DATETIME,              -- 比赛地本地时间
    league_id VARCHAR(50),                  -- 联赛统一编码
    league_name VARCHAR(100),               -- 联赛全称
    league_short_name VARCHAR(50),          -- 联赛简称
    season VARCHAR(20),                     -- 赛季: "2024-2025"
    match_round VARCHAR(20),                -- 轮次: "第18轮", "小组赛A组"
    
    -- 球队信息
    home_team_id VARCHAR(50),               -- 主队统一编码
    home_team_name VARCHAR(100) NOT NULL,
    home_team_short_name VARCHAR(50),
    away_team_id VARCHAR(50),               -- 客队统一编码
    away_team_name VARCHAR(100) NOT NULL,
    away_team_short_name VARCHAR(50),
    
    -- 状态与标识
    match_status TINYINT DEFAULT 0,         -- 0=未开始,1=进行中,2=完场,3=取消,4=延期
    is_neutral_venue BOOLEAN DEFAULT FALSE, -- 是否中立场地
    
    -- 审计字段
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    data_updated_at DATETIME,               -- 数据实际更新时间（从API获取的时间）
    
    -- 索引优化
    UNIQUE KEY idx_source_match (data_source, source_match_id),
    INDEX idx_match_time (match_time),
    INDEX idx_league (league_id, match_time),
    INDEX idx_teams (home_team_id, away_team_id),
    INDEX idx_status_time (match_status, match_time));
关键设计说明：

双ID系统：match_id(全局唯一) + internal_match_no(可读编号)


数据溯源：data_source + source_match_id + data_version 确保可追溯


时间双存储：UTC时间用于系统处理，本地时间用于展示


球队编码化：为后续球队维度分析做准备

第二层：比赛状态表 (match_status)
存储随时间变化的比赛状态信息
sql
CREATE TABLE match_status (
    status_id BIGINT PRIMARY KEY AUTO_INCREMENT,
    match_id CHAR(32) NOT NULL,
    
    -- 比赛阶段信息
    match_stage VARCHAR(30),                -- 比赛阶段: "pre_match", "first_half", "half_time", "second_half", "finished"
    match_minute INT,                       -- 比赛分钟数
    injury_time INT,                        -- 伤停补时分钟数
    
    -- 比分信息（支持多比分体系）
    home_score INT DEFAULT 0,               -- 主队得分
    away_score INT DEFAULT 0,               -- 客队得分
    home_ht_score INT,                      -- 主队半场得分
    away_ht_score INT,                      -- 客队半场得分
    home_ft_score INT,                      -- 主队全场得分（完场后确认）
    away_ft_score INT,                      -- 客队全场得分（完场后确认）
    
    -- 红黄牌
    home_yellow_cards INT DEFAULT 0,
    away_yellow_cards INT DEFAULT 0,
    home_red_cards INT DEFAULT 0,
    away_red_cards INT DEFAULT 0,
    
    -- 关键事件计数
    home_corners INT DEFAULT 0,             -- 角球
    away_corners INT DEFAULT 0,
    home_shots_on_target INT DEFAULT 0,     -- 射正
    away_shots_on_target INT DEFAULT 0,
    home_fouls INT DEFAULT 0,               -- 犯规
    away_fouls INT DEFAULT 0,
    
    -- 状态时间戳
    status_time DATETIME NOT NULL,          -- 状态记录时间
    is_latest BOOLEAN DEFAULT TRUE,         -- 是否为最新状态
    
    FOREIGN KEY (match_id) REFERENCES match_basic(match_id) ON DELETE CASCADE,
    INDEX idx_match_latest (match_id, is_latest),
    INDEX idx_status_time (status_time));
设计特点：

状态快照模式：每次更新插入新记录而非修改，完整保留比赛进程


多比分字段：区分实时、半场、终场比分


最新标识：通过is_latest快速获取当前状态

第三层：比赛结果表 (match_result)
比赛结束后的最终结果和结算信息
sql
CREATE TABLE match_result (
    result_id BIGINT PRIMARY KEY AUTO_INCREMENT,
    match_id CHAR(32) UNIQUE NOT NULL,
    
    -- 最终比分
    final_home_score INT NOT NULL,
    final_away_score INT NOT NULL,
    final_result CHAR(1) NOT NULL,          -- H=主胜, D=平, A=客胜
    
    -- 让球结果
    handicap_value DECIMAL(5,2),            -- 让球数
    handicap_result CHAR(1),                -- 让球盘结果
    
    -- 大小球结果
    over_under_line DECIMAL(5,2),           -- 大小球盘口
    over_under_result CHAR(1),              -- 大球/小球结果
    
    -- 半全场结果
    ht_result CHAR(1),                      -- 半场结果
    ft_result CHAR(1),                      -- 全场结果
    ht_ft_result CHAR(2),                   -- 半全场组合，如"HD"
    
    -- 进球时段分析
    goals_first_half INT DEFAULT 0,         -- 上半场总进球
    goals_second_half INT DEFAULT 0,        -- 下半场总进球
    goals_overtime INT DEFAULT 0,           -- 加时赛进球
    first_goal_minute INT,                  -- 首个进球分钟数
    first_goal_team CHAR(1),                -- 首球队伍: H/A
    
    -- 数据验证标志
    is_official_result BOOLEAN DEFAULT FALSE,  -- 是否为官方确认结果
    result_source VARCHAR(50),              -- 结果来源
    confirmed_at DATETIME,                  -- 结果确认时间
    
    FOREIGN KEY (match_id) REFERENCES match_basic(match_id) ON DELETE CASCADE,
    INDEX idx_result_type (final_result, handicap_result, over_under_result));
🔍 核心表字段设计的竞猜特征考量
1. 支持多种投注类型
sql
-- 在查询时快速计算各种玩法SELECT 
    -- 胜平负
    CASE WHEN final_home_score > final_away_score THEN 'H'
         WHEN final_home_score < final_away_score THEN 'A'
         ELSE 'D' END AS win_draw_loss,
    
    -- 让球胜平负
    CASE WHEN (final_home_score + handicap_value) > final_away_score THEN 'H'
         WHEN (final_home_score + handicap_value) < final_away_score THEN 'A'
         ELSE 'D' END AS handicap_result,
    
    -- 总进球数
    final_home_score + final_away_score AS total_goals,
    
    -- 半全场
    CONCAT(ht_result, ft_result) AS half_full_resultFROM match_result;

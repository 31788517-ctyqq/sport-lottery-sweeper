# 北单页面操作逻辑（开发版）

更新时间：2026-02-15  
适用页面：`/admin/beidan-filter`  
目标：将页面行为拆解为可实现、可测试、可验收的开发规则。

## 1. 范围与术语

### 1.1 页面模块
- 页头区：实时数据、规则入口、多策略入口
- 三维条件区：`ΔP`、`ΔWP`、`P-Tier`
- 其他条件区：联赛、期号、日期范围
- 策略区：策略下拉、策略列表、策略详情
- 策略管理区：生成当前策略、保存、管理
- 统计区：筛选统计数据
- 结果区：列表、排序、分页、导出、分析弹窗
- 多策略管理弹窗：定时策略配置与执行

### 1.2 核心术语
- 当前策略：用户基于三维条件点击“生成当前策略”后产生的临时策略（默认不持久化）。
- 已保存策略：用户命名保存到后端的策略。
- 示例策略：页面内置快捷条件模板（强势正路/冷门潜质/均衡博弈），仅用于快速填充条件。
- 策略生效：某策略被选中并触发筛选请求，统计区与结果区才展示数据。

## 2. 页面状态模型

### 2.1 基础状态
- `initialized`: 页面完成首轮初始化（策略列表、期号列表加载结束）。
- `strategySelected`: 是否有生效策略。
- `hasResults`: 本次筛选是否命中数据。
- `loading`: 当前是否在请求筛选接口。

### 2.2 显示规则
- 统计区、结果区默认隐藏。
- 仅当 `strategySelected=true` 后显示统计区与结果区。
- 若 `strategySelected=true` 且 `hasResults=false`，结果区展示空状态。

## 3. 初始化流程（页面加载/刷新）

1. 加载策略列表：`GET /api/v1/beidan-filter/strategies`
2. 加载可选期号：`GET /api/v1/beidan-filter/latest-date-times`
3. 默认选中最新期号（首个期号）
4. 自动执行一次筛选（使用当前表单默认条件）
5. 初始化完成，页面进入可交互态

实现要求：
- 任一步骤失败不应导致页面崩溃。
- 策略列表失败时允许用户继续使用三维+其他条件筛选。
- 期号接口失败时允许手动输入/保留空值并给出错误提示。

## 4. 操作逻辑（按模块）

## 4.1 获取实时数据（页头）

触发：点击“获取实时数据”  
行为：
1. 请求实时计数接口：`GET /api/v1/beidan-filter/real-time-count`
2. 更新页头“实时匹配 N 场”显示
3. 不改变当前筛选条件，不自动重新筛选

异常处理：
- 401/403/500/网络错误分别提示，不中断页面其他交互

## 4.2 三维条件与“当前策略”

触发：用户勾选/修改 `ΔP`、`ΔWP`、`P-Tier` 条件  
行为：
- 仅更新表单状态，不自动触发筛选。

触发：点击“生成当前策略”  
行为：
1. 校验三维条件是否为空
2. 非空时生成“当前策略”并加入策略列表首位（临时）
3. 自动选中“当前策略”
4. 触发筛选请求（见 4.6）

边界规则：
- 三维条件全空：不生成“当前策略”，提示用户先设置条件。
- 点击“重置”后：移除“当前策略”，统计区与结果区恢复隐藏。
- 页面刷新后：“当前策略”不保留（回归未选中状态）。

## 4.3 示例策略（强势正路/冷门潜质/均衡博弈）

触发：点击任一示例策略  
行为：
1. 将示例预设值填充到三维条件
2. 不自动触发筛选
3. 等待用户点击“生成当前策略”后生效

说明：
- 示例策略本质是“条件模板”，不是已生效策略，也不是持久化策略。

## 4.4 其他条件（联赛/期号/日期范围）

作用：
- 作为策略逻辑上的附加过滤条件，叠加在生效策略结果上。

规则：
- 仅修改其他条件不自动显示统计/结果。
- 当存在已生效策略时，变更其他条件后可触发重新筛选。
- 期号默认最新；用户可手动切换。

## 4.5 策略选择（下拉与卡片联动）

触发：
- 策略下拉选择某策略
- 点击策略卡片选择某策略

行为：
1. 双向同步选中状态（下拉与卡片）
2. 显示该策略详情
3. 触发筛选请求（见 4.6）
4. 将 `strategySelected` 置为 `true`

空态：
- 未选择策略时，策略详情区隐藏或显示“请选择策略”。

## 4.6 筛选执行（核心）

触发来源：
- 选中策略（当前策略或已保存策略）
- 已选策略前提下变更其他过滤条件
- 手动触发“应用筛选”（如页面有该动作入口）

请求：
- `POST /api/v1/beidan-filter/advanced-filter`

入参结构（建议）：
- `threeDimensional`: `powerDifference`, `winPanDifference`, `sizeBallDifference`
- `otherConditions`: `leagues`, `dateTime`, `dateRange`, 扩展条件
- `sort`: `field`, `order`
- `page`, `pageSize`

响应处理：
- 更新统计区数据
- 更新结果列表与分页
- 设置 `hasResults` 与总数

## 4.7 排序、分页、导出、分析

排序：
- 修改排序字段/排序顺序后，按新排序重新请求筛选接口。

分页：
- 页码/页大小变化触发重新请求，保持当前筛选条件不变。

导出：
- 支持 `Excel/CSV/JSON` 三种导出入口。
- 导出前校验：无结果时禁止导出并提示。
- 导出内容必须与当前筛选条件、排序、分页上下文一致（建议导出全量命中，不仅当前页）。

分析弹窗：
- 点击结果行“分析”打开 `AnalysisDialog`。
- 必须传递 `visible` 及对应比赛数据。
- 数据缺失时展示空态文案，不得报错。

## 4.8 策略管理（保存/修改/删除）

保存策略：
1. 基于当前三维条件与附加条件输入策略名
2. 调用 `POST /api/v1/beidan-filter/strategies`
3. 成功后刷新策略列表并可选中最新保存项

修改/删除策略：
- 在“管理策略”弹窗操作
- 修改成功后刷新列表并保持选中可用状态
- 删除选中策略后应回退到“未选中”或“当前策略”（按产品约定）

约束：
- 策略名去重校验
- 示例策略不可删改

## 4.9 多策略管理（独立子流程）

模块：`MultiStrategyManager`  
主要接口：
- `GET /api/multi-strategy/config?user_id={user}`
- `POST /api/multi-strategy/config`
- `POST /api/multi-strategy/execute`
- `POST /api/multi-strategy/toggle-task/{user_id}`
- `DELETE /api/multi-strategy/config/{task_id}`

规则：
- 401：提示登录态问题，不应持续刷错阻塞主页面。
- 500：展示可操作错误信息，允许用户重试。
- 多策略失败不影响北单单策略筛选主流程。

## 5. 事件触发矩阵（去重后）

会触发筛选请求的动作：
- 选择策略（下拉/卡片）
- 生成当前策略
- 已生效策略下修改其他条件
- 修改排序
- 修改分页

不会触发筛选请求的动作：
- 仅修改三维条件（未生成当前策略前）
- 点击示例策略（仅填充条件）
- 打开/关闭管理弹窗
- 获取实时数据

## 6. 错误处理规范

统一原则：
- 错误可见、可恢复、不中断主流程。
- 提示分级：鉴权（401）、权限（403）、服务异常（5xx）、网络异常。
- 避免吞错；控制台保留调试日志，界面展示简洁提示。

## 7. 发布验收清单（开发视角）

功能完整性：
- 初始化流程可稳定完成
- 当前策略生命周期符合规则（生成/生效/重置/刷新丢弃）
- 策略保存、修改、删除、复用闭环可用
- 排序/分页/导出/分析可用
- 多策略模块错误可控，不影响主流程

交互一致性：
- 统计区和结果区显示条件一致
- 下拉与策略卡片选中状态始终同步
- 空状态和错误状态提示准确

质量要求：
- 控制台无阻断错误（尤其是必填 props、未处理 Promise）
- 关键接口失败有兜底显示
- 自动化和手工测试通过（按项目测试清单）

## 8. 已知待完善项（用于后续开发）

- `AnalysisDialog` 仍有占位字段，需要后端返回完整分析数据模型。
- `Excel` 导出需确认是否为真实 `.xlsx` 二进制流，不应仅用 CSV 改扩展名。
- “应用降级规则”需确认是否已贯穿到后端筛选逻辑。
- 策略数据结构兼容分支（旧/新）建议统一，减少前端判断复杂度。
- 多策略接口鉴权与用户上下文需统一（避免 `user_id=admin` 硬编码导致 401/500）。

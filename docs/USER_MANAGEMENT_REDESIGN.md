# 用户管理改造建议文档需求说明

## 1. 背景与目标

### 1.1 背景
当前系统用户管理存在**前后台用户分离**的双轨设计：
- 后端有 `admin_user` 与 `user` 两套模型及相关 API（`admin_user_management.py`、`frontend_user_management.py`）。
- 前端有 `FrontendUsers.vue` 与 `BackendUsers.vue` 两个独立页面组件。
- 导致数据冗余、维护成本高、权限体系割裂，不符合传统 RBAC 用户管理最佳实践。

### 1.2 目标
- 建立**统一用户池 + RBAC + 扩展属性**的用户管理体系。
- 消除双轨冗余，实现**单表、单 API、单组件**。
- 满足**最小权限、职责分离、可追溯、良好体验**四大设计原则。
- 在后台左侧菜单中以**主菜单 + 子菜单**形式呈现完整用户管理功能，并与权限体系联动显隐。
- 为未来扩展数据权限、岗位、用户组等高级功能打好基础。

---

## 2. 核心设计原则

1. **最小权限原则**：用户仅拥有完成工作所需的最低权限。
2. **职责分离原则**：关键操作（权限分配、审计）由不同角色完成。
3. **可追溯原则**：所有关键操作记录日志，不可删除。
4. **用户体验优先**：后台操作清晰、高效、易用。

---

## 3. 统一用户管理体系架构

### 3.1 模型设计（RBAC + 扩展属性）
| 实体 | 说明 |
|------|------|
| **用户(User)** | 统一用户表，不区分前台/后台，通过角色关联权限范围。 |
| **角色(Role)** | 权限集合（管理员、运营、客服等），可多对多关联权限。 |
| **权限(Permission)** | 最小操作单元，细分：<br>① 页面/菜单权限<br>② 操作权限（增删改查、导出、审核）<br>③ 数据权限（部门/区域可见范围） |
| **部门(Department)** | 数据权限与用户分组的基础。 |
| **扩展属性** | 用户/角色可加额外字段（如积分、工号、岗位）。 |

### 3.2 数据库表（核心）
- `users`（id, username, email, phone, password_hash, status, department_id, role_ids, …）
- `roles`（id, name, description, is_system）
- `permissions`（id, name, resource_type, action, parent_id）
- `user_roles`、`role_permissions`、`departments`
- `operation_logs`（操作人、对象、内容、IP、时间）

### 3.3 API 设计（RESTful）
| 方法 | 路径 | 功能 |
|------|------|------|
| GET | `/api/v1/users` | 列表 + 多条件筛选 |
| POST | `/api/v1/users` | 创建用户 |
| GET | `/api/v1/users/{id}` | 详情 |
| PUT | `/api/v1/users/{id}` | 修改信息/角色/部门 |
| DELETE | `/api/v1/users/{id}` | 禁用（逻辑删除） |
| GET | `/api/v1/roles` | 角色列表 |
| PUT | `/api/v1/roles/{id}` | 修改角色权限 |
| GET | `/api/v1/permissions` | 权限树 |
| GET | `/api/v1/departments` | 部门树 |
| GET | `/api/v1/audit-logs` | 操作日志 |

## 4. 功能模块设计

### 4.1 用户管理
- 列表 & 查询：多条件筛选、导出。
- 用户操作：增、删（禁用）、改、查、状态切换、重置密码。
- 支持批量操作与 Excel 导入（严格校验）。

### 4.2 角色与权限管理
- 角色增删改查（系统内置不可删）。
- 权限树（父子联动选择）。
- 角色授权（勾选方式）。
- 权限复制。

### 4.3 部门/组织架构管理
- 树形结构展示。
- 部门增删改移、负责人设置。

### 4.4 个人中心与安全
- 修改个人信息（需验证）。
- 安全设置：改密、MFA、登录设备/历史。
- 个人操作日志查看。

### 4.5 操作日志审计
- 记录敏感操作（权限变更、角色分配、密码重置、状态变更）。
- 操作人、对象、内容、IP、时间，管理员不可删除。

---

## 5. 高级与安全功能
- **登录审计与风控**：记录 IP、地点、设备、成败状态，异常报警与锁定。
- **会话管理**：管理员可查看并强制下线用户会话。
- **数据权限引擎**：根据角色+部门动态过滤数据。
- **批量操作 & 导入/导出**。

---

## 6. 后台菜单规划

### 6.1 主菜单
- **用户管理**  
  路径：`/admin/users`  
  图标：`fas fa-users`  
  权限：`user:read`

### 6.2 子菜单结构
| 子菜单 | 路径 | 功能 | 权限 |
|--------|------|------|------|
| 用户列表 | `/admin/users/all` | 查看、搜索、筛选、导出、启用/禁用、分配角色/部门、重置密码 | `user:read` / `user:write` |
| 角色与权限 | `/admin/users/roles` | 角色管理、权限树、授权、权限复制 | `role:read` / `role:write` |
| 部门管理 | `/admin/users/departments` | 组织架构树、部门增删改移、负责人设置 | `department:read` / `department:write` |
| 个人中心 | `/admin/users/profile` | 当前用户信息、安全设置、登录历史、个人日志 | `profile:read` / `profile:write` |
| 操作日志 | `/admin/users/audit-logs` | 查看所有用户关键操作日志 | `audit:read` |

### 6.3 菜单实现方式
- **推荐**：SidebarMenu 支持二级菜单（嵌套路由），动态根据权限过滤显隐。
- 备选：一级菜单 + 页面内 Tab 切换（改动小，但层级感弱）。

---

## 7. 权限控制方案
- 每个菜单项绑定权限标识（如 `user:read`）。
- 全局权限校验方法 `hasPermission(perm)` 控制菜单显隐与页面访问。
- 页面内按钮级权限可结合指令或条件渲染实现。

---

## 8. 技术实现要点
- **前端**：Vue3 + Element Plus，`@/views/admin/users/` 目录下分模块组件（UserList、RolePermission、DepartmentManagement、UserProfile、AuditLogs）。
- **后端**：统一 Service `UnifiedUserManager`，权限校验在网关/中间件层统一处理。
- **缓存**：用户权限信息 Redis 缓存，变更时刷新。
- **密码安全**：bcrypt / Argon2，禁止明文。

---

## 9. 分阶段实施路线

| 阶段 | 目标 | 关键任务 | 产出 |
|------|------|----------|------|
| 阶段1 | 统一数据模型与API | 合并用户表、角色表；合并 API；前端单组件 | 单表、单 API、单组件 |
| 阶段2 | RBAC 权限体系接入 | 建权限表、实现权限树与角色授权 UI；后端中间件鉴权 | 可分配角色权限，页面/操作受控 |
| 阶段3 | 数据权限 & 高级功能 | 部门表 & 数据权限引擎；登录审计、操作日志、MFA；批量导入导出 | 完整安全与高效管理闭环 |

---

## 10. 验收标准
- 前端左侧菜单按规划显示用户管理及其子菜单，权限不足时不显示。
- 所有用户相关操作均记录日志且不可删除。
- 角色与权限可灵活配置，并实时生效。
- 支持按部门过滤数据。
- 密码安全存储，支持 MFA 登录保护。
- 批量导入用户数据格式校验通过率 100%。

---

## 11. 风险与应对
- **风险**：迁移期间可能影响现有用户登录。  
  **应对**：阶段1保持旧字段兼容，灰度切换。
- **风险**：权限配置错误导致功能不可用。  
  **应对**：内置系统管理员角色不可删，提供权限回滚机制。

---

## 12. 结语
本方案从业务、技术、权限、用户体验多维度规划了用户管理模块的改造路径，既能解决当前双轨制的痛点，又能支撑未来业务复杂化。建议按阶段推进，确保平稳过渡。
